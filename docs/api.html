1. Overview

Here is the CUTe’s programmatic API reference manual. If you need an in-depth description of CUTe usage, you should refer to the User guide instead.

2. Build configuration

At Build configuration time, multiple build options are available to customize final CUTe build. From client code, you may eventually refer to the corresponding C macros listed below:

  • CONFIG_CUTE_INTERN_ASSERT

  • CONFIG_CUTE_UTEST

3. Functional domains

3.1. Test definition

Various C preprocessor macros are exposed to define test cases. These are:

  • CUTE_TEST

  • CUTE_TEST_DEFN

  • CUTE_TEST_DECL

  • CUTE_TEST_STATIC

  • CUTE_TEST_EXTERN

See test case definition section of User guide for more informations.

3.2. Fixture operations

Various C preprocessor macros are exposed to attach test fixture functions to tests or suites at definition time. These are:

  • CUTE_NULL_SETUP

  • CUTE_NULL_TEARDOWN

  • CUTE_INHR_SETUP

  • CUTE_INHR_TEARDOWN

See fixture operations section of User guide for more informations.

3.3. Timeouts

Various C preprocessor macros are exposed to assign timeouts to tests or suites at definition time. These are:

  • CUTE_DFLT_TMOUT

  • CUTE_INHR_TMOUT

  • CUTE_NONE_TMOUT

See timer section of User guide for more informations.

3.4. Group definition

Various C preprocessor macros are exposed to define test groups. These are:

  • CUTE_REF

  • CUTE_GROUP

  • CUTE_GROUP_DEFN

  • CUTE_GROUP_STATIC

  • CUTE_GROUP_EXTERN

3.5. Suite definition

Various C preprocessor macros are exposed to define test suites. These are:

  • CUTE_SUITE

  • CUTE_SUITE_STATIC

  • CUTE_SUITE_EXTERN

See suite definition section of User guide for more informations.

3.6. Test assertion

Multiple test assertions are exposed to implement testing logic. These are described in the following sections.

You may also refer to Test assertion section of User guide for more informations.

3.6.1. Basic assertion

cute_check_assert implements a basic versatile test assertion exposed as a C preprocessor macros.

4.2. Macros

4.2.1. CUTE_CONFIG_INIT

CUTE_CONFIG_INIT

Test run configuration default initializer.

Initialize a cute_config with default values such that a test hierarchy runner behaves as following :

  • test case and suite nodes are all enabled,

  • debug mode is off,

  • CUTE_CONFIG_TERSE_REPORT is enabled,

  • CUTE_CONFIG_PROBE_TTY is enabled.

4.2.2. CUTE_DFLT_TMOUT

CUTE_DFLT_TMOUT

Default test timeout.

timeout in seconds assigned to test when not explicitly specified. It is 3 seconds long.

See also

  • CUTE_INHR_TMOUT

  • CUTE_NONE_TMOUT

4.2.3. CUTE_FLT_RANGE

CUTE_FLT_RANGE(_min, _max)

Define a floating point number range.

Initialize a cute_flt_range variable that defines a range of floating point numbers.

Example 

CUTE_TEST(mytest)
{
     const struct cute_flt_range range = CUTE_FLT_RANGE(-5.5f, 20.5f);

     cute_check_flt_range(0.5f, in,     range);
     cute_check_flt_range(10,   not_in, CUTE_FLT_RANGE(-5.5, 5.5));
}

See also

  • cute_check_flt_range

  • cute_flt_range

Parameters:

  • _min – [in] range’s minimum value (inclusive)

  • _max – [in] range’s maximum value (inclusive)

4.2.4. CUTE_FLT_SET

CUTE_FLT_SET(...)

Define a floating point number set.

Initialize a cute_flt_set variable that defines a set of floating point numbers.

The ... variable argument list is an unordered list of floating point values separated by commas that composes the set.

Example 

CUTE_TEST(mytest)
{
     const struct cute_flt_set set = CUTE_FLT_SET(-5.5f, 0.5f, 5.5f);

     cute_check_flt_set(0.5f, in,     set);
     cute_check_flt_set(7,    not_in, CUTE_FLT_SET(-10.5, 0.5, 10.0));
}

See also

  • cute_check_flt_set

  • cute_flt_set

Parameters:

  • ... – [in] list of set values

4.2.5. CUTE_GROUP

CUTE_GROUP(_name)

Define a test group with default class storage.

Define a test group designated by the _name variable that may later be used to define suites. The _name test group variable is defined with default class storage specifier, i.e., static global file scope.

CUTE_GROUP must be immediately followed by an initializer as stated in CUTE_GROUP_DEFN.

Example 

CUTE_TEST(sample_test_0)
{
     cute_check_assert(0 == 0);
}

CUTE_TEST(sample_test_1)
{
     cute_check_assert(1 == 1);
}

CUTE_GROUP(sample_group) = {
     CUTE_REF(sample_test_0),
     CUTE_REF(sample_test_1)
};

CUTE_SUITE(sample_suite, sample_group);

See also

  • CUTE_REF

  • CUTE_GROUP_STATIC

  • CUTE_GROUP_EXTERN

  • CUTE_GROUP_DEFN

Parameters:

  • _name – [in] test group name

4.2.6. CUTE_GROUP_DEFN

CUTE_GROUP_DEFN(_name)

Define a test group.

Define a test group designated by the _name variable that may later be used to define suites.

CUTE_GROUP_DEFN must be immediately followed by an initializer. It should define the collection of tests (and / or suites) that composes the _name group thanks to the CUTE_REF macro as shown in the example below.

Example 

CUTE_TEST(sample_test_0)
{
     cute_check_assert(0 == 0);
}

CUTE_TEST(sample_test_1)
{
     cute_check_assert(1 == 1);
}

static CUTE_GROUP_DEFN(sample_group) = {
     CUTE_REF(sample_test_0),
     CUTE_REF(sample_test_1)
};

CUTE_SUITE(sample_suite, sample_group);

See also

  • CUTE_REF

  • CUTE_GROUP

  • CUTE_GROUP_STATIC

  • CUTE_GROUP_EXTERN

  • CUTE_GROUP_DECL

  • CUTE_SUITE

Parameters:

  • _name – [in] test group name

4.2.7. CUTE_GROUP_DECL

CUTE_GROUP_DECL(_name)

Declare a test group.

Declare a test group designated by the _name variable that may later be used to define suites.

Use this in combination with CUTE_GROUP_EXTERN or CUTE_GROUP_DEFN to allow referencing defined test group from multiple compilation units.

Example 

#ifndef _SAMPLE_TEST_H
#define _SAMPLE_TEST_H

#include <cute/cute.h>

extern CUTE_GROUP_DECL(sample_group);

#endif

See also

  • CUTE_GROUP_DEFN

  • CUTE_GROUP_EXTERN

Parameters:

  • _name – [in] test group name

4.2.9. CUTE_GROUP_EXTERN

CUTE_TEST(mytest)

     cute_check_bool(true, is, true);
}

Ensure that true unequals false : 

CUTE_TEST(mytest)
{
     cute_check_bool(true, is_not, false);
}

4.2.37. cute_check_flt

cute_check_flt(_chk, _op, _xpct)

Check a floating point number.

Abort current test and mark it as failed if the comparison of _chk and _xpct using the _op comparison operator results in a failure. Comparison is performed according to the following formula : 

_chk <_op> _xpct
Where _op MUST be one of :

  • equal to ensure that _chk == _xpct ;

  • unequal to ensure that _chk != _xpct ;

  • greater, to ensure that _chk > _xpct ;

  • greater_equal, to ensure that _chk >= _xpct ;

  • lower, to ensure that _chk < _xpct ;

  • lower_equal, to ensure that _chk <= _xpct.

Both _chk and _xpct MUST be floating point numbers, i.e., either :

  • float,

  • double,

  • long double,

  • or an equivalent typedef’ed type.

This macro may be used from within fixture functions as well as test case functions.

Ensure that 0.5 equals 0.5 (single precision) : 

CUTE_TEST(mytest)
{
     cute_check_flt(0.5f, equal, 0.5f);
}

Ensure that 0.5 unequals 1.5 (single precision) : 

CUTE_TEST(mytest)
{
     cute_check_flt(0.5f, unequal, 1.5f);
}

Ensure that 1.5 is greater than 0.5 (double precision) : 

CUTE_TEST(mytest)
{
     cute_check_flt(1.5, greater, 0.5);
}

Ensure that 1 is greater than or equal to 1 (double precision) : 

CUTE_TEST(mytest)
{
     cute_check_flt(1.5, greater_equal, 1.5);

4.2.81. cute_expect_uint_set

cute_expect_uint_set(_func, _parm, _op, _xpct)

Schedule an unsigned integer set function parameter mock expectation.

Define and schedule an expectation that is checked when cute_mock_uint_parm() is called.

The check performed aborts current test and marks it as failed if the comparison of _parm against the _xpct set using the _op comparison operator results in a failure. Comparison is performed according to the following formula : 

_parm <_op> _xpct
Where _op MUST be one of :

  • in, to ensure that _parm equals to one the _xpct set values ;

  • not_in to ensure that _parm equals to none of the _xpct set values.

_parm MUST be an unsigned integer, i.e., either :

  • unsigned char,

  • unsigned short,

  • unsigned int,

  • unsigned long,

  • unsigned long long,

  • or an equivalent typedef’ed type.

_xpct MUST be a cute_uint_set unsigned integer set as defined by the CUTE_UINT_SET macro.

This macro may be used from within fixture functions as well as test case functions.

Example 

#include <cute/cute.h>
#include <cute/expect.h>
#include <limits.h>

static void
callee(int word, short half)
{
     cute_mock_uint_parm(word);
     cute_mock_uint_parm(half);
}

static void
caller(int word, short half)
{
     callee(word, half);
}

CUTE_TEST(mytest)
{
     const struct cute_uint_set set = CUTE_UINT_SET(0, 10, USHRT_MAX / 2);

     cute_expect_uint_set(callee, word, in,     CUTE_UINT_SET(0, 5, 10));
     cute_expect_uint_set(callee, half, not_in, set);
     caller(10, USHRT_MAX);
}

See also

  • cute_mock_uint_parm()

  • CUTE_UINT_SET

  • cute_uint_set

  • CUTE_TEST

Parameters:

  • _func – [in] function name

  • _parm – [in] function parameter name

  • _op – [in] constraint operation used to check the expectation

  • _xpct – [in] reference unsigned integer set to perform the check against

const char *match

Test hierarchy matching regular expression.

Allows to select a subset of the test hierarchy to run.

At run initialization time, each node of the test hierarchy is compared against the cute_config::match regular expression and selected for running if matched. Nodes for which the match failed are disabled.

cute_config::match must be specified as a POSIX extended regular expression as described in regex(7) . cute_config::icase alters the matching process.

cute_config::match may also be specified as NULL in which case, no matching is performed, meaning that the complete test hierarchy is selected for running.
